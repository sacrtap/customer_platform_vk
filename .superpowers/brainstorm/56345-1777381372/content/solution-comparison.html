优化方案对比

选择最适合当前项目的修复方案

方案 A：最小修复（推荐）

只修复 Drawer 中操作后不刷新详情的问题 + 补充后端取消接口

优点

  • 改动最小，风险最低
  • 直接解决用户反馈的问题
  • 后端补充 cancel 接口，完善 API
  • 不影响现有业务逻辑

缺点

  • 没有操作反馈动画
  • 失败时没有乐观更新
修改清单

后端 (2 处)

  • billing.py Service: 新增 cancel_invoice() 方法
  • billing.py Route: 新增 POST /invoices/:id/cancel 路由

前端 (2 处)

  • billing.ts API: 新增 cancelInvoice() 函数
  • Invoices.vue: 修改 handleConfirm/handleCancel/handleSubmit/handleComplete 调用详情接口

方案 B：增强用户体验

方案 A + 操作反馈动画 + 乐观更新 + 错误回滚

优点

  • 用户体验更流畅
  • 操作即时反馈（loading 状态）
  • 失败时自动回滚状态

缺点

  • 代码复杂度增加
  • 需要处理更多边界情况
  • 开发时间更长

方案 C：状态管理重构

引入 Pinia store 管理结算单状态，全局共享状态

优点

  • 状态管理更规范
  • 多组件共享状态更方便
  • 便于后续扩展

缺点

  • 改动范围大
  • 引入新的状态管理层
  • 当前项目可能过度设计
