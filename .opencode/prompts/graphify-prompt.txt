你拥有 OpenCode 原生 graphify 工具（7 个内置工具和MCP服务）
图谱数据位于 graphify-out/graph.json。

【工具清单与场景】
| 问题类型 | 使用工具 | 示例 |
| 调用链 / 依赖 / 模块结构 | graphify_query_graph (mode=bfs) | "认证流程涉及哪些模块？" |
| A 概念如何到达 B 概念 | graphify_shortest_path | "Customer 如何关联到 Invoice？" |
| 某函数/类的完整信息 | graphify_get_node | "AnalyticsService 的定义" |
| 某节点的直接连接 | graphify_get_neighbors | "谁调用了 BalanceService？" |
| 某个社区/模块的全部节点 | graphify_get_community | "结算模块包含什么？" |
| 系统最核心的模块 | graphify_god_nodes | "核心抽象是什么？" |
| 图谱整体统计 | graphify_graph_stats | "图谱规模如何？" |
| 影响范围评估 / 开发前调研 | graphify_query_graph + graphify_god_nodes | "新增客户导出功能会影响哪些模块？" |

【决策规则】
问题涉及调用关系 / 依赖 / 模块结构 / 影响范围 / 概念关系 → 使用 graphify 工具
问题涉及具体实现细节 / 逐行逻辑 / bug 修复 → 读取源代码

【图谱覆盖范围】
- 代码：backend/ + frontend/ 全部源文件
- 文档：docs/ 目录已纳入图谱（含概念节点与设计规格文档）
- 文档与代码节点之间存在交叉连接，可通过图谱查询文档与代码的关联
- 如需查询文档内容，优先尝试图谱工具；图谱信息不足时再直接读取文件

【开发前影响评估】
接到新功能或变更需求时，先用图谱评估影响范围：
1. 用 graphify_query_graph 从相关概念/模块出发，BFS 探索关联节点
2. 用 graphify_god_nodes 识别核心模块，判断是否触及系统关键路径
3. 用 graphify_get_neighbors 查看直接依赖方，列出需同步变更的模块
4. 结合图谱结果拆解开发任务，确保覆盖所有影响面

【图谱更新】
代码或文档变更后，图谱可能过期。如用户提及最近修改了内容或代码文件有变更后，自动执行：
/graphify --update
如果上述命令执行失败，再使用以下bash命令执行
python3 -c "from graphify.watch import _rebuild_code; from pathlib import Path; _rebuild_code(Path('.'))"

【输出要求】
1. 说明使用的工具及查询目标
2. 基于图谱结构回答，优先信任图谱中的显式连接
3. 引用 source_file / source_location 定位源码或文档
4. 图谱信息不足时明确说明，再 fallback 读代码
