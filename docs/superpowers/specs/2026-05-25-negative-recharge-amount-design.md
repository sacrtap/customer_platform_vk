# 负数金额充值支持设计

**日期**: 2026-05-25
**状态**: 已审批

## 背景

当前余额管理页面（`/billing/balances`）的充值功能仅支持正数金额。业务需要支持双向资金流动——同一表单既能充值（正数）也能扣款（负数），且赠送金额也需支持负数。

## 决策记录

| 决策 | 选择 |
|------|------|
| 实充金额范围 | C：支持正数和负数 |
| 赠送金额范围 | D：支持正数和负数，无限制 |
| 额度限制 | 不限额，负数可随意使用 |
| UX 术语 | A：保留"充值"标签，用正负号区分方向 |
| 交互方案 | B：最小改动 + 负数二次确认 |

## 架构

### 前端（Balance.vue）

1. **移除输入限制**：
   - `real_amount` 的 `a-input-number` 移除 `:min="0.01"`，允许输入负数
   - `bonus_amount` 的 `a-input-number` 移除 `:min="0"`
   - placeholder 更新为提示负数含义

2. **修改提交校验**：
   - 移除 `real_amount <= 0` 的阻断校验
   - 新增 `real_amount === 0` 时阻断（不允许零金额）
   - 当 `real_amount < 0` 时弹出确认对话框（Modal.confirm），显示扣除总金额

3. **二次确认对话框**：
   - 文案："本次操作将从客户余额中扣除 XXX 元，是否确认？"
   - 确认按钮："确认扣款"
   - 取消则中断流程

### 后端（billing.py）

1. **路由校验调整**（`routes/billing.py:378`）：
   - 将 `real_amount <= 0` 改为 `real_amount == 0`
   - 禁止零金额，允许正负数
   - bonus_amount 暂无额外校验（与实充行为一致）

2. **Service 层无需修改**（`services/billing.py:56-94`）：
   - 现有逻辑 `balance.real_amount += real_amount` 天然支持负数减法
   - RechargeRecord 创建逻辑不依赖金额符号

### 数据库

- 不做修改：`DECIMAL(12, 2)` 字段已允许负数
- 不添加 `CheckConstraint`

### 测试影响

- **需更新**：断言"充值金额必须 > 0"的测试用例
- **新增**：
  - 负数充值场景：余额 10000 → 扣款 3000 → 余额 7000
  - 赠送金额负数场景
  - 二次确认交互（前端 E2E）
  - 零金额阻断测试

## 不变部分

- API 接口签名不变（字段名、类型保持一致）
- 权限控制不变（`billing:recharge` 权限）
- 数据库模型不变
- Service 层核心逻辑不变

## 风险

| 风险 | 缓解措施 |
|------|----------|
| 误操作扣款 | 二次确认对话框 |
| 余额扣为负数 | 允许欠费（业务决策），可通过余额列表展示 |
| API 绕过前端 | 后端校验禁止零金额，允许正负数（与前端一致） |
