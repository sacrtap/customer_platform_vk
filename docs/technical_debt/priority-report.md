## 三、建议（2026-07-05 更新，三选一）

### 选项 A：后端 TD-001 拆分（2-3 周，中风险）
1. **先拆 `backend/app/routes/billing.py`（1913 行）**，按子功能拆分为 invoices/payments/recharge 子路由；
2. **再拆 `backend/app/services/analytics.py`（1301 行）**，分离报表计算/数据聚合逻辑；
3. **三拆 `backend/app/services/billing.py`（1007 行）**，提取公共事务/状态机逻辑；
4. 拆分时需同步补充结算事务边界测试，确保状态机逻辑一致性。

### 选项 B：前端 TD-002 收尾（1-2 周，低风险，可与 A 并行）
1. **拆 `frontend/src/views/customers/Detail.vue`（1217 行 → ≤500 行）**，复用已有 4 个 customers 组件；
2. **拆 `frontend/src/views/customers/Index.vue`（1038 行 → ≤500 行）**，提取表格/分页/筛选为 composables；
3. 完成后 TD-002 完全解除，前端仅余通用组件目录维护。

### 选项 C：A+B 并行推进（3-4 周，资源允许时）
- 前后端分别派生子代理并行执行，互不阻塞；
- 前端 TD-002（低风险）先完成后，资源可集中到 TD-001。

- **TD-008/009**：低优先级，已有部分基础设施。

## 五、决策建议（2026-07-05 更新）

- **推荐选项 B（最高置信度）**: 先完成前端 TD-002 剩余拆分（1-2 周，低风险），完全解除前端债务后再评估 TD-001。
